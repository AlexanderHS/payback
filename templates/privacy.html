Privacy

Plain English. Honest about what we can and can't see. The whole codebase is on GitHub; verify any of this for yourself.

What we store

  • Your Google email address — it's your account ID.
  • Everything you type into your projects (name, target, strategy, costs, benefits, notes, all of it).
  • The hashed form of any API keys you create. The plaintext key is shown to you once at creation and never stored.
  • The emails of users you've shared with, so we can let them view your projects.
  • Server logs (request paths, response codes, timestamps, IP) — kept for a few weeks, used only for debugging.

What we don't store

  • Passwords. Google handles authentication; we never see one.
  • Third-party tracking, analytics, or advertising cookies. There are no scripts on this site beyond the app itself.
  • Plaintext API keys. (Hashed at rest; verify via db.py.)

Who can see your data

  • You, when signed in.
  • Anyone you've explicitly shared with, via the Sharing page. Read-only, one-way, revocable any time.
  • The site administrator (currently the project author) has shell access to the server and, in principle, can read the SQLite database. We don't have client-side encryption, so this is a real capability — not a theoretical one. We're being honest about that because the alternative would be to make a cryptographic claim we can't actually keep while also supporting the AI-agent API workflow this is built around.

Operationally, the admin commits to:

  • Not querying user project content except for technical debugging (e.g. you reported a problem you want fixed) or to comply with a legal order.
  • Treating the site as a hobby project that should leave no fingerprint on your life beyond what you explicitly put into it.
  • Keeping admin tools minimal — the /api/analytics endpoint, for example, returns aggregate counts and storage sizes only, never project content. See get_analytics() in db.py.

If that operational assurance isn't enough — for example, if you're a business that needs the data physically on your own infrastructure — run your own instance. The stack is small (one Flask container plus an oauth2-proxy sidecar, ~1,400 lines of Python). Bring your own Google OAuth credentials and a domain; the README has the docker-compose setup. We don't see anything you do with it.

API keys and AI agents

If you create an API key and give it to an AI agent or third-party tool, that key has full read and write access to your projects. The agent's host (OpenAI, Anthropic, whichever) will see whatever you let the agent send and receive on your behalf, and may retain it under their own policies. Treat an API key like a password.

Sharing

When you share with someone, we record their email. They'll see your projects the next time they sign in to Payback with that Google account. Sharing is read-only and you can revoke any time from the Sharing page.

Deletion

Projects: archive, then permanently delete from the Archived tab. Account: open a GitHub issue or contact the author through the address in the repo's commit history, and we'll wipe your row and any shares pointing at it.

Security

  • HTTPS everywhere (terminated at the reverse proxy).
  • API keys hashed (SHA-256) at rest.
  • CSRF tokens on all state-changing web requests.
  • API rate limiting per key.
  • Per-user data caps to prevent runaway storage abuse.
